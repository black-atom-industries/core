## Goals

1. Replace `culori`-based color utilities (`oklch()`, `tint()`) with `@colordx/core` chainable color objects
2. Split feedback and git colors into dedicated nested properties on `ui`
3. Get the monitor app working end-to-end with one proof-of-concept theme before mass-migrating

## Scope

**This plan covers:** Phase 1 only â€” introducing `ThemeDefinitionNext` (colors-grouped, nested ui), building the `createThemeDefinition()` helper, and getting one "next" theme file rendering in the monitor alongside the legacy themes.

**Out of scope (future plans):**

- Mass-migrating the other ~50 themes to colordx
- Removing `culori` from deps
- Adapter templates: adapters opt in to themes via their own `black-atom-adapter.json`, so the new `-next` theme will not be auto-generated by any adapter. Adapter template updates for the nested shape happen in Phase 2/3.

## Strategy decision: introduce `ThemeDefinitionNext` with a `colors:` namespace

Two changes ride together:

1. **Structural â€” group colors under `colors:`.** `ThemeDefinitionNext` separates metadata from the color tree:

   ```ts
   interface ThemeDefinitionNext {
       meta: ThemeMeta;
       colors: {
           primaries: ThemePrimaryColors;
           palette: ThemePaletteColors;
           ui: ThemeUiColorsNext; // nested feedback / git
           syntax: ThemeSyntaxColors;
       };
   }
   ```

   This makes `walk()` cleaner â€” it traverses `input.colors` only, never crossing into `meta`. The `instanceof Colordx` leaf check stays (it differentiates Colordx from structural objects within the colors tree); meta is preserved verbatim by `createThemeDefinition` and not touched by `walk()`.

2. **`ui` shape â€” nested `feedback` / `git`.** `ThemeUiColorsNext` drops the legacy flat keys (`negative`, `warn`, `info`, `hint`, `positive`, `add`, `delete`, `modify`) in favor of `feedback: ThemeFeedbackColors` and `git: ThemeGitColors` on both `bg` and `fg`.

`ThemeDefinition` (legacy, top-level, flat ui) and `ThemeDefinitionNext` (colors-grouped, nested ui) coexist during transition. `themeMap` is typed `Record<ThemeKey, ThemeDefinition | ThemeDefinitionNext>`. Phase 2 migrates the 33 legacy themes to `ThemeDefinitionNext`. Phase 3 deletes the legacy type.

**Why two types instead of widening one?** Top-level vs. colors-grouped is a structural difference, not a per-key one â€” a single optional union would not narrow well in consumers and would defeat the readability win of the namespace. Two distinct types with a discriminated union (`"colors" in theme`) are cleaner.

**CSS variable contract is preserved.** `themeToCssVars` flattens from inside the colors subtree (`flatten(theme.colors)`), so paths produce the same CSS var names (`--ba-primaries-d10`, `--ba-ui-bg-default`, â€¦). The `colors` segment is for TS code organization, not CSS.

#### 1.2 Compatibility shim in `themeToCssVars`

The CSS var generator now needs to (a) read from `theme.colors` for new themes and (b) emit legacy flat aliases for the new nested shape until consumers are updated.

```ts
export function themeToCssVars(
    theme: ThemeDefinition | ThemeDefinitionNext,
): Record<string, string> {
    if ("colors" in theme) {
        const vars = flatten(theme.colors);
        // Legacy flat aliases for transitional CSS â€” removed in Phase 2.2
        for (const scope of ["bg", "fg"] as const) {
            const feedback = theme.colors.ui[scope].feedback;
            const git = theme.colors.ui[scope].git;
            if (feedback) {
                vars[`--ba-ui-${scope}-negative`] = feedback.negative;
                vars[`--ba-ui-${scope}-warn`] = feedback.warning;
                vars[`--ba-ui-${scope}-info`] = feedback.info;
                vars[`--ba-ui-${scope}-positive`] = feedback.success;
            }
            if (git) {
                vars[`--ba-ui-${scope}-add`] = git.add;
                vars[`--ba-ui-${scope}-delete`] = git.delete;
                vars[`--ba-ui-${scope}-modify`] = git.modify;
            }
        }
        return vars;
    }
    const { meta: _, ...rest } = theme;
    return flatten(rest);
}
```

Notes:

- `hint` and the structural keys are not aliased â€” they have no nested counterpart.
- Path prefix is identical for both branches (`flatten` starts inside the color tree), so `--ba-ui-bg-default` etc. work for both legacy and new themes.
- The shim is single-purpose, time-bounded, and removed in Phase 2.2.

#### 1.3 Build `walk()` and `createThemeDefinition()` helper

Create `src/lib/create-theme-definition.ts`:

- Takes a theme-like object and a callback, applies the callback via a generic `walk()` helper
- `walk()` traverses every node (objects, arrays, primitives) and calls the callback in **post-order** (children first, then parent)
- The callback decides what to do with each value â€” `walk()` itself knows nothing about colors or `toHex`
- Callback is explicit so it can be swapped later for different output formats (union type)

**Treat `Colordx` as a leaf, not a walkable object.** `isWalkable()` MUST early-out on `instanceof Colordx` so traversal does not descend into Colordx internals. Without this, post-order would build a Colordx-shaped plain object (no prototype methods) and the parent callback would receive that â€” `toHex` would crash.

**Callback API: single object argument.** Drop the two-positional form; pass `{ node, path }` so future fields (e.g. `parent`, `key`) can be added without breaking callers.

**Typing strategy.** The callback transforms types (`Colordx â†’ string`), so input type â‰  output type. Type the input as `unknown` / a generic walkable, type the return as `unknown`, and assert the final return type at the `createThemeDefinition` callsite. The earlier `WalkNode<T>` recursive type was abandoned â€” it produced a non-narrowing union and fought inference once transforms were introduced.

```ts
import { Colordx } from "@colordx";

type WalkKey = string | number | symbol;
type WalkPath = readonly WalkKey[];

interface WalkContext {
    node: unknown;
    path: WalkPath;
}

type WalkCallback = (ctx: WalkContext) => unknown;

function isWalkable(value: unknown): value is Record<PropertyKey, unknown> {
    if (typeof value !== "object" || value === null) return false;
    if (value instanceof Colordx) return false;
    if (value instanceof Date) return false;
    if (value instanceof RegExp) return false;
    return true;
}

export function walk(input: unknown, callback: WalkCallback): unknown {
    function visit(node: unknown, path: WalkPath): unknown {
        if (Array.isArray(node)) {
            const next = node.map((item, index) => visit(item, [...path, index]));
            return callback({ node: next, path });
        }

        if (isWalkable(node)) {
            const next: Record<PropertyKey, unknown> = {};
            for (const key of Reflect.ownKeys(node)) {
                next[key] = visit(node[key], [...path, key]);
            }
            return callback({ node: next, path });
        }

        return callback({ node, path });
    }

    return visit(input, []);
}
```

Usage in `createThemeDefinition`:

```ts
import { Colordx } from "@colordx";
import type { ThemeDefinitionNext, ThemeMeta } from "../types/theme.ts";

interface ThemeAuthoringInput {
    meta: ThemeMeta;
    colors: {
        primaries: ThemeRawPrimaryColors;
        palette: ThemeRawPaletteColors;
        ui: ThemeUiColorsNextRaw; // Colordx-bearing nested shape
        syntax: ThemeSyntaxColorsNextRaw; // Colordx-bearing
    };
}

export function createThemeDefinition(input: ThemeAuthoringInput): ThemeDefinitionNext {
    return {
        meta: input.meta,
        colors: walk(input.colors, ({ node }) => {
            if (node instanceof Colordx) return node.toHex();
            return node;
        }) as ThemeDefinitionNext["colors"],
    };
}
```

`walk()` only traverses `input.colors`; meta is preserved verbatim. Inside the colors subtree, every leaf is either a Colordx (converted) or a primitive (passed through). The `instanceof Colordx` check is what differentiates leaves from structural objects.

**Reuse:** `ThemeDefinitionNext` (return type) and the `ThemeRaw*` / `ThemeUiColorsNextRaw` / `ThemeSyntaxColorsNextRaw` interfaces (input shape) all defined in 1.1. `Colordx` type and class from `@colordx`.

**TDD approach:** Write tests first, implementing `walk()` incrementally â€” one behavior at a time (primitive passthrough â†’ plain object recursion â†’ array handling â†’ Colordx-as-leaf conversion â†’ nested mixed structures). See `dev-style-tdd` skill for the red-green-refactor loop.
